DNS records cheat sheet

A practical field guide for spotting what each DNS record does, which records can safely be repeated, and which mistakes break websites, email, certificates, or verification checks.

1
User asks for a name
Browser, mail server, app, or validation tool looks up a hostname.
2
DNS returns records
The answer depends on the exact name, type, and authoritative DNS host.
3
The service is found
Traffic, mail, certificates, and checks follow the records that were returned.

Read a DNS record

Every DNS change starts with four fields. Be exact with the hostname and record type before adding or replacing anything.

Name

Hostname being looked up, such as www, @, _dmarc, or selector1._domainkey.

Type

The kind of answer requested, such as A, AAAA, MX, TXT, or CNAME.

Value

The returned answer: an IP address, hostname, mail route, policy, verification token, or text value.

TTL

Lower TTL = faster updates once existing caches have expired. Lower TTL before planned changes.

Common record types

Use this as the quick scan before making a change. The repeatability label means what is normal at the same hostname.

AMultiple OK

IPv4 address

Websites, servers, VPN portals, RDS gateways.

Example: www points to 203.0.113.10.

AAAAMultiple OK

IPv6 address

IPv6-enabled websites and services.

IPv6 version of an A record.

CNAMEOne only

Alias to another name

SaaS validation, provider-managed targets, and www aliases.

A CNAME must not coexist with almost any other record at the same name. Points to a hostname, not an IP.

MXMultiple OK

Inbound email route

Microsoft 365, Google Workspace, Mimecast, Proofpoint.

Lower number = higher priority. Do not mix unrelated mail providers.

TXTMultiple OK

Plain text information

SPF, DMARC, domain verification, security tokens.

A container record. SPF and DMARC have special rules within TXT.

SPFOne per host

Approved email senders

Defines which servers may send email for the domain.

Starts with v=spf1. Two SPF records cause a permanent error. Merge into one.

DKIMMulti selector

Email signing key

Microsoft 365, marketing platforms, CRM systems.

DKIM is usually published as TXT, but some providers use CNAME records that point to provider-managed DKIM keys.

DMARCOne per target

Anti-spoofing policy

Controls what receivers should do when neither SPF nor DKIM produces an aligned pass for the visible From domain.

Lives at _dmarc.example.com. Multiple records can cause the policy to be ignored.

NSMultiple OK

Nameserver authority

DNS hosting and subdomain delegation.

Changing NS can move DNS control. Escalate if unsure.

CAAMultiple OK

Certificate authority control

Controls which CAs may issue SSL certificates for the domain.

Incorrect CAA can break certificate renewal.

PTRUsually one

Reverse DNS lookup

Mail server reputation, reverse lookups, logging.

Controlled by the IP owner, usually the ISP or cloud provider.

SRVMultiple OK

Service location

SIP, Teams or Skype legacy, LDAP, autodiscover.

Includes protocol, priority, weight, port, and target hostname.

SOAOne per zone

Start of authority

Core zone management and timing settings.

Managed by the DNS host. Rarely edited by 1st line.

ALIASProvider varies

Root-domain alias

Root domain pointing to CDNs or hosted platforms.

Not a universal DNS standard. Often called ANAME or CNAME flattening.

Email security stack

Email DNS is layered. MX handles inbound delivery; SPF, DKIM, and DMARC help receivers decide whether mail claiming to be from the domain should be trusted.

MX

Receive mail

Where should inbound email for this domain be delivered?

SPF

Authorise senders

Which servers are allowed to send mail for this domain?

DKIM

Prove message integrity

Was this message signed by an approved mail system and unchanged in transit?

DMARC

Set anti-spoofing policy

What should receivers do when no aligned SPF or DKIM pass exists for the visible From domain?

DMARC policy values

p=noneMonitor only. Collect reports first.
p=quarantineTreat failing mail as suspicious.
p=rejectReject failing mail entirely.

SPF endings

The ending controls how strongly the domain owner describes unlisted senders.

-allHard fail. Only listed senders pass. Strongest.
~allSoft fail. Unlisted senders are suspicious. Common during setup.
?allNeutral. No strong statement. Weak protection.
+allAllow all. Dangerous. Avoid unless absolutely necessary.

Common DNS mistakes

These are the checks most likely to prevent avoidable outages or failed mail authentication.

X

Two SPF records

SPF returns a permanent error and mail may be rejected.

Fix: merge both into one v=spf1 record.
X

Two DMARC records

Receivers may discard the DMARC policy entirely.

Fix: merge into one DMARC TXT record at _dmarc.example.com.
X

Mixing unrelated MX providers

Mail may route to the wrong system unpredictably.

Fix: use only the intended mail-routing design.
X

CNAME alongside other records

A CNAME cannot share the same hostname with normal service records.

Fix: use the provider-recommended type or a different hostname.
X

MX pointing to a CNAME

MX targets must not point to a CNAME. They should point to a hostname with A and/or AAAA records.

Fix: point MX to a real mail host with address records.
X

Editing NS records without checking

Changing NS records can move live DNS away from the correct provider.

Fix: confirm nameserver ownership and migration plan first. Escalate if unsure.
X

Assuming DNS changes are instant

Cached records remain until TTL expires, so propagation takes time.

Fix: check TTL and retest only after the cache has expired.

Basic troubleshooting flow

A repeatable path for ticket evidence and safer DNS changes.

Identify the affected service

Website not loading: check A, AAAA, CNAME. Email not arriving: check MX. Junk mail: check SPF, DKIM, DMARC, PTR.

Confirm the DNS host

Check the domain's NS records. Make the change only where the authoritative zone is managed.

Check the exact hostname

Compare example.com, www.example.com, _dmarc.example.com, and selector names carefully.

Check for conflicts

Look for an existing CNAME, SPF record, DMARC record, or old MX records from a previous provider.

Record evidence

Capture old value, new value, date and time changed, TTL, ticket reference, and customer approval where required.

Test after the change

Use Resolve-DnsName, nslookup, admin portal checks, mail header analysis, and validation tools.

Golden rules

Apply these before making any DNS change.

One SPF record per hostname. Merge, never duplicate.
One DMARC record per DMARC target. Merge, never duplicate.
Multiple DKIM selectors are normal. Each service can get its own.
Multiple MX records are normal when part of one mail-routing design.
CNAME records should stand alone at their hostname.
NS changes can move the whole DNS zone. Escalate if unsure.
Always check the current record before adding a new one.
Keep before-and-after evidence in the ticket.
Standards reference: SPF RFC 7208, DKIM RFC 6376, DMARC RFC 9989, core DNS RFC 1034 and RFC 1035, DNS clarifications RFC 2181, IPv6 AAAA RFC 3596, CAA RFC 8659.
